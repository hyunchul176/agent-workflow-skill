1소개

이 가이드는 Claude Code에서 여러 역할의 에이전트를 운용하는 방법을 설명합니다. 혼자 진행하는 프로젝트라도 기획·실행·검토를 역할별로 나누면 실제 팀처럼 작업 흐름을 관리할 수 있습니다. 별도 앱을 새로 설치하는 구조가 아니라, Claude가 따르는 규칙 파일과 프로젝트 폴더 안의 상태 파일을 함께 사용하는 방식입니다.

무엇을 해주나

  • 프로젝트를 네 가지 기본 역할로 나눕니다. main은 조율, planner는 계획, executor는 실행, reviewer는 검토를 맡습니다.
  • 메시지 앞에 역할 이름을 적으면(예: executor), Claude가 해당 역할의 책임과 기준에 맞춰 작업합니다.
  • 역할별 기억 파일을 따로 유지해, 대화를 새로 열어도 이전 진행 상황을 이어갈 수 있습니다.
  • 목표 · 계획 · 할 일 · 기억 · 활동 기록이 모두 프로젝트 폴더 안의 파일로 남습니다.
  • 별도 서버나 SSH 연결 없이, 로컬 프로젝트 폴더만으로 동작합니다.
› executor, phase 1 step 1 구현해줘
# 메시지 앞에 역할 이름을 적으면 그 역할의 책임과 기억을 기준으로 작업합니다.
main planner executor reviewer + 추가 역할

2왜 좋은가

🧩

한 사람이 여러 역할을 나눠 맡을 수 있습니다

기획·구현·검토를 같은 사람이 하더라도 역할을 분리해 진행합니다. 각 역할의 책임이 명확해 작업이 섞이거나 빠질 가능성이 줄어듭니다.

🧠

이전 대화를 이어갈 수 있습니다

역할마다 MEMORY.md가 있어, 새 대화에서도 어떤 일을 했고 다음에 무엇을 해야 하는지 확인할 수 있습니다.

📡

작업 흐름이 기록으로 남습니다

작업이 끝날 때마다 activity_log.md에 요약이 남습니다. 나중에 누가 무엇을 바꿨는지 쉽게 되짚을 수 있습니다.

🔁

수정과 검토를 반복할 수 있습니다

executor와 reviewer가 수정과 테스트를 최대 다섯 번까지 반복해, 단순 구현에서 끝나지 않고 검증까지 이어집니다.

그냥 AI와 대화할 때와 무엇이 다른가

장점이 한 번의 짧은 작업에서는 잘 드러나지 않습니다. 차이는 작업이 길어지고 대화가 여러 번 이어질 때 나타납니다.

맥락 유지 (이어가기·되짚기)
그냥 AIclaude -c·-r로 이어갈 수 있지만, 대화 기록을 통째로 불러오는 방식입니다. 길어지면 자동 요약돼 세부가 누락되고, 과거 결정을 다시 찾으려면 긴 로그를 뒤져야 합니다.
워크플로우진행 상태와 결정·근거가 압축된 파일(기억·활동 기록·노트)로 남습니다. 어느 새 대화에서든 읽어 이어가고, 대화 길이와 상관없이 바로 되짚을 수 있습니다.
‘다 됐다’의 신뢰도
그냥 AI만든 주체가 스스로 완료를 선언합니다. 검증이 느슨해도 겉으로 드러나지 않습니다.
워크플로우만든 역할은 완료 표시를 못 합니다. reviewer가 따로 테스트하고 planner가 확인한 뒤에야 ✅가 붙습니다.
여러 사람이 함께
그냥 AI사람마다 시키는 방식과 결과물 형식이 제각각이라, 다른 사람의 진행을 이어받기 어렵습니다.
워크플로우역할·규칙·상태 파일이 저장소에 함께 있어, 모두가 같은 방식으로 일하고 결과물 형식도 통일됩니다. 한 사람이 멈춘 작업을 다른 사람이 그대로 이어받습니다.

대신 역할을 지정하고 파일을 유지하는 약간의 수고가 듭니다. 그래서 짧은 일에는 과할 수 있습니다.

언제 쓰면 좋은가

모든 작업에 네 역할이 필요한 것은 아닙니다. 이 워크플로우는 작업 기간이 길거나, 구현과 검토가 반복될 때 특히 유용합니다.

적합한 경우
  • 기획, 구현, 검토가 여러 차례 반복되는 기능 개발
  • 대화창을 닫았다 다시 열며 며칠에 걸쳐 이어가는 프로젝트
  • 구현자와 검토자의 관점을 분리해 품질을 높이고 싶을 때
  • 결정 과정과 변경 이력을 나중에 다시 확인해야 하는 작업
직접 시키는 게 나은 경우
  • 파일 하나만 고치는 짧은 작업
  • 몇 분 안에 끝나는 간단한 질문
  • 기록을 남길 필요가 없는 일회성 실험

3맥락을 어떻게 유지하나

핵심은 간단합니다. 진행 상황을 AI가 알아서 기억하는 것이 아니라, 프로젝트 폴더 안의 파일에 남겨 두는 방식입니다.

각 역할은 작업을 시작하기 전에 정해진 파일을 먼저 읽습니다. 그래서 대화를 새로 열어도 목표, 계획, 할 일, 지난 작업 기록을 다시 불러올 수 있습니다. 기억은 모델 내부에만 남는 것이 아니라 .workflow 폴더의 마크다운 파일에 저장됩니다. 이 절차는 CLAUDE.md에 적혀 있고, Claude는 매번 그 규칙을 기준으로 움직입니다.

맥락을 담는 파일들

📋
목표와 계획project_brief · roadmap · todos
프로젝트의 목적, 단계별 흐름, 지금 해야 할 일을 담습니다. planner가 주로 관리합니다.
🧠
역할별 기억MEMORY.md
각 역할이 어디까지 진행했는지, 다음에 무엇을 이어 해야 하는지 기록합니다.
📡
공용 활동 기록activity_log.md
모든 역할이 작업 후 남기는 공용 일지입니다. 전체 작업 흐름을 확인할 때 사용합니다.
🚧
막힌 일blockers.md
당장 해결하지 못한 문제, 담당자, 필요한 후속 조치를 기록합니다. 해결되면 목록에서 정리합니다.

역할을 부르면 일어나는 일

1
필요한 파일을 먼저 읽습니다. 역할 정의와 프로젝트 상태를 정해진 순서로 확인합니다.내 역할(ROLE) → 목표 → 계획 → 할 일 → 막힌 일 → 내 기억 → 공용 기록
2
현재 상태를 짧게 정리합니다. 지금까지 진행된 내용과 이번에 할 일을 먼저 알려줍니다.
3
역할에 맞는 작업을 수행합니다. planner는 계획을 정리하고, executor는 구현하며, reviewer는 검토합니다.
4
필요하면 공용 기록에 남깁니다. 파일을 만들거나 바꾼 경우 activity_log.md에 요약을 추가합니다.
5
다음에 이어갈 내용을 저장합니다. 자기 역할의 MEMORY.md에 결과와 다음 작업을 갱신합니다.
가벼운 질문이나 잡담처럼 파일 변경이 없는 경우에는 기록과 기억 갱신을 생략합니다. 불필요한 흔적을 남기지 않기 위해서입니다.

4네 개의 역할

각 역할의 책임과 넘겨야 할 일은 ROLE.md에 정해져 있습니다. 그래서 Claude가 역할을 혼동하지 않고, 계획·실행·검토를 분리해 처리할 수 있습니다.

역할은 이렇게 이어집니다

요청하고 싶은 일을 전달합니다.
무엇을 누가 할지 정함▼
main요청을 알맞은 역할에 배분하고 전체 흐름을 살핍니다.
계획 요청▼
planner목표를 단계(phase)와 할 일(todo)로 나눕니다.
할 일 전달▼
executor할 일을 구현하고 결과물을 만듭니다.
⇄loop
최대 5회
reviewer테스트하고 문제와 수정 방향을 보고합니다.
“확인 부탁”▼
planner결과물을 직접 확인한 뒤 완료(✅)로 표시합니다.
↺ 남은 할 일이 있으면 executor 단계로 돌아가 같은 흐름을 반복합니다.
네 역할은 따로 도는 것이 아니라, 모두 .workflow 폴더의 같은 상태 파일을 읽고 씁니다. 한 역할의 결과가 파일에 남고 다음 역할이 그 파일을 읽기 때문에, 작업이 자연스럽게 이어집니다.

각 역할이 맡는 일

main노트 작성 ✗

조율자: 사용자의 의도를 파악하고, 어떤 역할이 맡을지 정합니다.

  • 요청을 알맞은 역할로 연결
  • 전체 흐름과 막힌 일 확인
  • 직접 기획·구현·검토하거나 완료 표시하지 않음
planner노트 작성 ✗

계획자: 목표를 단계와 할 일로 쪼개고, 완료 여부를 최종 확인합니다.

  • 목표를 단계(phase)로 나누고, 각 단계를 실행 가능한 할 일로 정리
  • 직접 확인한 뒤에만 할 일에 완료(✅) 표시
  • 구현은 executor에게, 테스트는 reviewer에게 넘김
executor노트 작성 ✓

실행자: 할 일 목록에 적힌 작업을 실제 코드나 결과물로 만듭니다.

  • 코드 작성, 파일 수정, 결과물 생성
  • 중요한 파일을 덮어쓰기 전 백업
  • 작업을 끝내도 스스로 완료 표시하지 않고 “확인 부탁”으로 보고
reviewer노트 작성 ✓

검토자: 결과물을 테스트하고, 문제와 수정 방향을 정리합니다.

  • 테스트 작성·실행, 통과/실패와 재현 방법 보고
  • 자동 반복이 끝나면 변경 사항과 남은 이슈 정리
  • 실제 수정 작업은 executor에게 넘김

프로젝트 성격에 따라 역할을 더 만들 수 있습니다. /agent-workflow add-agent <이름>을 사용하면 됩니다. 예를 들어 연구 프로젝트라면 research · data_analysis · writing, 제품 개발이라면 designer · qa 같은 역할을 추가할 수 있습니다.

프로젝트에 맞춰 역할 조합하기

기본 4역할(main · planner · executor · reviewer)은 항상 있습니다. 역할은 부를 때만 동작하므로, 안 쓰는 역할은 따로 뺄 필요 없이 부르지 않으면 됩니다. research처럼 필요한 역할만 추가합니다. * 표시는 직접 추가한 역할입니다.

💻 소프트웨어 개발
plannerexecutorreviewer

기본 역할만으로 충분합니다. 버그가 있으면 loop로 구현과 검토를 반복합니다.

🔬 연구 · 문헌조사
plannerresearch*writing*reviewer

research가 자료를 모으고 노트로 정리하며, writing이 초고를 작성합니다. reviewer는 논리와 사실관계를 점검합니다.

✍️ 글쓰기 · 문서
plannerexecutorreviewer

executor가 초고를 작성하고, reviewer가 문장과 사실관계를 검토합니다. 참고 자료나 결정 사항은 노트로 남깁니다.

📊 데이터 분석
plannerdata_analysis*reviewer

data_analysis가 분석과 시각화를 맡고, reviewer가 코드 실행 결과와 해석이 재현되는지 확인합니다.

5모드와 명령

역할은 main · planner · executor · reviewer 네 가지뿐입니다. 여기서 다루는 ask · loop · note는 역할이 아니라 모드로, 역할이 일하는 방식을 바꿉니다. 특히 loop는 새 역할이 아니라 executor와 reviewer를 자동으로 번갈아 돌리는 모드입니다. 키워드가 없으면 기본 실행으로 처리됩니다.

역할을 부르는 형식은 정해져 있지 않습니다. 메시지 앞에 역할 이름이 분명히 드러나면 됩니다(executor, … / you are executor 모두 동작). 역할을 적지 않고 하고 싶은 일만 말해도 됩니다. Claude가 요청을 보고 알맞은 역할을 골라 “executor로 진행합니다”처럼 밝힌 뒤 작업합니다. 동작 방식은 CLAUDE.md에서 바꿀 수 있습니다.
모드이렇게 부르면이렇게 동작합니다
기본 실행<역할>, …역할의 파일을 읽고, 작업하고, 필요한 경우 기록과 기억을 갱신합니다.
가벼운 질문<역할> ask …역할의 맥락만 참고해 답합니다. 파일이나 기억은 수정하지 않습니다.
구현·검토 자동 반복… loop …executor가 수정하고 reviewer가 테스트하는 과정을 최대 5회 반복합니다. 테스트가 통과하면 자동으로 멈춥니다.
노트로 남기기<역할> note …알아낸 내용이나 결정 사항을 정해진 양식의 노트 파일로 저장합니다. 노트 작성 권한이 있는 역할만 수행합니다.
막힌 일 등록대화 중 확인 후지금 해결할 수 없는 문제를 담당자, 우선순위, 필요한 조치와 함께 기록합니다. 해결되면 정리합니다.
완료 확인planner verify …planner가 결과물을 직접 확인한 뒤 할 일에 완료(✅) 표시를 합니다.

바로 복사해서 쓸 수 있는 예시

› planner, 목표를 단계와 할 일로 정리해줘
› executor, phase 1 step 1을 구현해줘
› reviewer, executor가 만든 결과를 테스트해줘
› executor ask, 이 함수는 어떤 방식으로 캐싱되고 있어?   # 파일을 건드리지 않는 질문
› loop, 로그인 모듈의 동시성 버그를 고쳐줘                # 모드: executor↔reviewer 자동 반복
› reviewer note, 부하가 몰리면 커넥션 풀 잠금 문제로 가끔 500 오류가 발생함
› planner verify phase 1 step 1

6활용 예시

자율주행 평가를 위한 적대적 시나리오 생성 연구를 예로, 한 사이클이 어떻게 진행되는지 보여줍니다. 카드 왼쪽 아바타와 색으로 어느 역할인지 구분하고, 카드 사이 화살표는 한 역할의 결과가 다음 역할로 넘어가는 흐름을 나타냅니다.

💬 예시 대화 · 적대적 시나리오 생성 연구
📋
🧑‍💻 나→planner
planner, 자율주행 평가용 적대적 시나리오 생성 실험을 설계하려고 해. 단계별로 계획을 잡아줘
phase 1을 ① 베이스라인 cut-in 생성기, ② 피험 시스템(SUT) 반응 기반 적대적 교란, ③ 충돌률 평가 지표로 나눴습니다. todos.md에 정리했습니다.
planner가 만든 todos를 executor에게 전달

executor, phase 1 step 1(베이스라인 cut-in 생성기)을 구현해줘

이 예시의 핵심은 대화를 새로 열어도 역할이 이전 진행 상황을 복원한다는 점입니다. 3번 ‘맥락을 어떻게 유지하나’에서 설명한 것처럼, 작업을 시작하기 전에 관련 파일을 먼저 읽기 때문입니다.

7지켜야 할 규칙

✅ 완료 표시는 planner만

작업을 수행한 역할은 스스로 완료 처리하지 않습니다. “확인 부탁” 상태로 보고하고, planner가 결과물을 직접 확인한 뒤 ✅ 표시를 합니다. 이 규칙은 ✅가 실제 검증을 의미하도록 만들기 위한 것입니다.

＋ 기록은 덮지 않고 추가

활동 기록, 막힌 일, 노트는 기존 내용을 지우지 않고 새 줄로 덧붙입니다. 그래야 지난 판단과 변경 이력이 사라지지 않습니다. 너무 길어지면 오래된 기록만 별도로 정리합니다.

@ 어느 역할로 일하는지는 항상 분명히

역할마다 책임과 기억이 분리돼 있어, 어느 역할로 일하느냐가 결과를 좌우합니다. 역할을 직접 지정하면 그대로 따르고, 지정하지 않으면 Claude가 요청을 보고 알맞은 역할을 고른 뒤 “executor로 진행합니다”처럼 밝히고 작업합니다. 모호하면 먼저 확인합니다.

¶ 노트는 수정하지 말고 새로 남기기

한 번 작성한 노트는 당시 판단의 기록으로 남깁니다. 내용이 바뀌면 새 노트를 작성하고, 기존 노트에는 ‘대체됨’ 표시만 남깁니다.

8시작하기

처음 도입할 때는 아래 세 단계만 진행하면 됩니다. 한 번 설정해 두면 이후에는 프로젝트 안에서 계속 같은 워크플로우를 사용할 수 있습니다.

1
스킬 폴더 복사: .claude/skills/agent-workflow/ 폴더를 프로젝트에 복사합니다. 모든 프로젝트에서 쓰려면 ~/.claude/skills/에 두면 됩니다.
2
프로젝트 초기화: 해당 프로젝트에서 /agent-workflow init을 실행합니다. 그러면 .workflow/ 폴더와 규칙 파일(CLAUDE.md)이 만들어지고, 프로젝트 목표를 한 줄로 입력받습니다.
3
역할로 시작: 이제 planner, 목표를 정리해줘처럼 역할 이름을 앞에 적어 대화하면 됩니다.

폴더 구조

초기화가 끝나면 프로젝트 안에 아래와 같은 파일들이 만들어집니다. 모두 일반 마크다운 파일이라 직접 열어 읽고 수정할 수 있습니다.

your-project/
├── CLAUDE.md                        # 워크플로우 규칙 (init이 설치)
├── .claude/skills/agent-workflow/   # 스킬 본체
└── .workflow/
    ├── shared/                      # 모두가 공유하는 상태
    │   ├── project_brief.md         # 목표와 배경
    │   ├── roadmap.md               # 단계별 큰 흐름 · planner 관리
    │   ├── todos.md                 # 구체적인 할 일 · planner 관리
    │   ├── blockers.md              # 막힌 일
    │   └── activity_log.md          # 공용 활동 기록
    └── agents/<역할>/
        ├── ROLE.md                  # 역할별 책임과 넘길 일
        ├── MEMORY.md                # 역할별 기억
        └── notes/                   # 노트
